# ChuHanAI 阶段一 PRD（联网中国象棋）

## 1. 文档目标
本 PRD 用于指导 ChuHanAI 阶段一（联网对战）研发落地，仅覆盖在线匹配、实时对弈、聊天与基础对局功能，不包含阶段二（自主训练 AI）。

## 2. 背景与目标
项目目标是提供一个可直接打开即玩的中国象棋联网对战客户端，保证规则正确、对局流畅、跨平台可用（macOS/Windows）。

### 2.1 成功标准（阶段一）
1. 用户启动客户端后可在 10 秒内进入匹配队列。
2. 匹配成功后可稳定进入对局，支持完整规则判定。
3. 支持悔棋、认输、求和、计时、聊天等基础能力。
4. 核心规则测试覆盖率达到高标准（见测试章节）。

## 3. 技术约束与边界
- **语言与版本**：Java（JDK 21）
- **核心技术栈**：Spring Boot + MySQL + Netty + JavaFX
- **不做项**：
  - 不做注册/登录
  - 不做段位/ELO 复杂匹配
  - 不做 AI 自主训练

## 4. 竞品与开源参考（调研结论）
以下项目用于功能与架构参考（不要求完全复刻）：

1. [springboot 中国象棋在线对战](https://github.com/13213824850/chess)：对战服务端分层与房间管理可借鉴。  
2. [chessonline_springboot](https://github.com/Qapi/chessonline_springboot)：在线对局流程组织可参考。  
3. [Javafx-ChineseChess](https://github.com/gyc514893758/Javafx-ChineseChess)：JavaFX 棋盘交互、悔棋、计时实现思路可参考。  
4. [联机象棋（Netty）](https://github.com/zjx1230/chess)：客户端/服务端长连接通信模型可参考。  
5. [Pikafish](https://github.com/official-pikafish/Pikafish)、[ElephantEye](https://github.com/xqbase/eleeye)：仅用于规则校验思路参考，不纳入阶段一功能。  

调研结论：阶段一优先采用“服务端权威规则判定 + 客户端渲染”的模式，避免双端规则不一致导致争议。

## 5. 用户画像与核心场景
### 5.1 用户画像
- 无需注册、希望快速开局的轻量玩家。
- 期望对弈过程稳定、规则准确、界面清晰。

### 5.2 核心使用路径
1. 打开客户端 -> 进入大厅。
2. 点击“开始匹配” -> 等待匹配成功。
3. 进入对局 -> 下棋 + 聊天。
4. 使用悔棋/求和/认输等功能。
5. 对局结束 -> 查看结果 -> 返回大厅继续匹配。

## 6. 功能需求（MVP）
### 6.1 客户端（JavaFX）
1. **主界面布局**
   - 左侧：棋盘 + 棋子 + 对局状态（红黑方、回合方、计时）。
   - 右侧：聊天室（系统消息 + 双方聊天）。
2. **棋盘交互**
   - 选中棋子高亮可走位置。
   - 非法走子本地提示，不发送或发送后由服务端驳回并回滚。
3. **对局控制**
   - 悔棋申请（对方同意后生效）
   - 求和申请（对方同意后和棋）
   - 认输（立即结束）
   - 重连恢复（基础版：重连后恢复当前局面）
4. **资源要求**
   - 棋子与棋盘图片资源需可商用或开源可用（需记录来源与许可证）。

### 6.2 服务端（Spring Boot + Netty）
1. **匹配系统（简化）**
   - 单池 FIFO 匹配策略。
   - 优先保证可用性，不引入分段位池。
2. **对局房间管理**
   - 创建房间、分配红黑方、广播状态。
   - 维护回合、步数、计时、局面快照。
3. **规则引擎（服务端权威）**
   - 校验每一步是否合法。
   - 校验将军、绝杀、困毙、和棋条件（阶段一最低支持：将死、认输、求和）。
4. **实时通信协议**
   - 连接建立、心跳保活、消息序列号、断线重连、幂等处理。
5. **聊天系统**
   - 房间内双人聊天，限制消息长度与频率。

### 6.3 计时系统
- 基础时间：每方 10 分钟。
- 步进加时：每走一步 +15 秒。
- 计时由服务端统一推进并广播，客户端仅展示。
- 超时判负：服务端判定并结束对局。

## 7. 中国象棋规则范围（阶段一必须）
### 7.1 棋子走法
必须完整支持：将/士/象/马/车/炮/兵（卒）走法与限制（如蹩马腿、塞象眼、炮架、将帅不可照面）。

### 7.2 对局结束判定
1. 将死（无合法解将手段）
2. 一方认输
3. 双方求和达成
4. 超时判负

### 7.3 悔棋规则
- 悔棋仅支持“悔一步”（双向协商制）。
- 悔棋成功后回退局面、回合与计时状态（按实现策略可回退至上一步快照）。

## 8. 非功能需求
1. **性能**
   - 匹配响应：P95 < 2s（低并发阶段）
   - 走子确认：P95 < 300ms（同城网络）
2. **稳定性**
   - Netty 长连接支持心跳与断线检测。
   - 服务端异常后可快速恢复（基础日志 + 错误兜底）。
3. **兼容性**
   - 客户端必须通过 macOS 与 Windows 双平台运行验证。
4. **安全与风控（基础）**
   - 输入校验、防止非法消息注入、聊天限流。

## 9. 系统设计（高层）
### 9.1 模块划分
1. `client-ui`：JavaFX 页面与本地状态机
2. `gateway-netty`：实时通信接入层
3. `match-service`：匹配服务
4. `game-service`：房间与规则引擎
5. `chat-service`：聊天消息处理
6. `infra-mysql`：对局记录与日志落库

### 9.2 数据模型（最小集合）
1. `match_queue`：排队信息（匿名会话ID、入队时间）
2. `game_room`：房间信息（房间ID、红黑方会话ID、状态）
3. `game_move`：走子记录（步号、from/to、棋子类型、时间戳）
4. `game_result`：结果信息（胜负、结束原因、耗时）
5. `chat_message`：房间聊天消息

## 10. 通信协议（建议）
### 10.1 消息类型
- `MATCH_JOIN` / `MATCH_SUCCESS`
- `MOVE_REQUEST` / `MOVE_ACCEPTED` / `MOVE_REJECTED`
- `UNDO_REQUEST` / `UNDO_RESPONSE`
- `DRAW_REQUEST` / `DRAW_RESPONSE`
- `RESIGN`
- `CHAT_SEND` / `CHAT_BROADCAST`
- `TIME_SYNC`
- `GAME_OVER`
- `PING` / `PONG`

### 10.2 协议要求
- 每条消息包含：`msgId`、`roomId`、`sessionId`、`timestamp`、`payload`。
- 关键写操作消息（走子、认输等）需要幂等校验（`msgId` 去重）。

## 11. 测试策略（重点）
### 11.1 规则测试（必须高覆盖）
1. 每个棋子走法正反用例（合法/非法）。
2. 特殊限制用例（蹩马腿、象不过河、将帅照面等）。
3. 将军与解将路径全覆盖。
4. 胜负判定与计时判负测试。

### 11.2 联机流程测试
1. 匹配 -> 开局 -> 落子 -> 结束全链路。
2. 悔棋、求和、认输流程测试。
3. 断线重连恢复与消息重放测试。

### 11.3 跨平台测试
- macOS 与 Windows：
  - 界面渲染一致性
  - 输入事件（鼠标点击、拖拽）一致性
  - 打包启动流程与依赖完整性

## 12. 里程碑计划（建议）
1. **M1（第1周）**：基础工程搭建、协议骨架、JavaFX 棋盘渲染。
2. **M2（第2周）**：匹配、房间、基础走子同步打通。
3. **M3（第3周）**：规则引擎完善 + 计时 + 悔棋/求和/认输。
4. **M4（第4周）**：测试加固、跨平台联调、发布候选版。

## 13. 验收标准（DoD）
1. 能稳定完成至少 100 局内部对战无严重规则错误。
2. 规则核心自动化测试全部通过，关键分支覆盖充分。
3. macOS/Windows 客户端均可安装运行并完成完整对局。
4. 已知问题仅剩低优先级 UI/体验项，不影响对局主流程。

## 14. 风险与应对
1. **规则复杂导致 bug**：采用“服务端唯一判定 + 快照回放 + 高密度单元测试”。
2. **跨平台 UI 差异**：早期双平台并行验证，不在最后阶段集中处理。
3. **网络抖动导致状态不一致**：引入心跳、序列号、状态重同步机制。

## 16. 推荐工程结构（包与模块）
建议采用多模块 Maven 工程（或 Gradle 多项目），确保边界清晰。

### 16.1 模块结构建议
1. `chuhanai-common`：DTO、枚举、错误码、协议常量。
2. `chuhanai-server-gateway`：Netty 接入、编解码、会话管理。
3. `chuhanai-server-match`：匹配队列、配对逻辑。
4. `chuhanai-server-game`：房间管理、规则引擎、计时器。
5. `chuhanai-server-chat`：聊天限流与广播。
6. `chuhanai-server-infra`：MySQL DAO、缓存、日志。
7. `chuhanai-client-javafx`：UI、交互、网络客户端。

### 16.2 `game` 模块包结构建议
- `domain.board`：棋盘、棋子、坐标模型
- `domain.rule`：规则校验器（按棋子拆分）
- `domain.state`：局面状态、回合、计时
- `application.command`：走子/悔棋/认输命令处理
- `application.query`：局面快照查询
- `infrastructure.repository`：房间与走子持久化

## 18. AI 编程执行规范（强制）
本章节定义后续 AI 编程阶段的研发流程，作为强制标准执行。

### 18.1 小模块定义
“小的完整模块”必须同时满足：
1. 功能边界清晰（例如：马走法校验、匹配入队、悔棋申请流程）。
2. 可独立编译并有明确输入/输出。
3. 可被单元测试覆盖核心逻辑。

### 18.2 强制流程（每个模块都执行）
1. 拆分模块任务并定义完成标准（DoD）。
2. 先写/补单元测试用例（正常+异常+边界）。
3. 实现模块代码。
4. 运行该模块相关单元测试并确保全部通过。
5. 自检代码风格与静态检查。
6. 仅在测试通过后提交一次 commit。

### 18.3 Commit 准入规则
不满足以下条件，禁止提交：
1. 模块测试未全部通过。
2. 引入新的失败测试或明显回归。
3. 功能未达到模块 DoD。

### 18.4 Commit 粒度与信息规范
1. 一个 commit 只做一个小模块，不混入无关修改。
2. commit message 建议格式：
   - `feat(module): ...` 新功能
   - `fix(module): ...` 缺陷修复
   - `test(module): ...` 补充测试
3. message 要说明“为什么做”，不仅是“改了什么”。

### 18.5 推荐执行节奏
1. 规则引擎相关模块优先（正确性最高优先级）。
2. 每完成 1 个模块即执行“测试 -> commit”闭环。
3. 每完成 3~5 个 commit，执行一次全量回归测试。

### 18.6 质量红线
1. 规则判定逻辑禁止仅靠手工点击验证，必须有自动化单测。
2. 涉及计时、悔棋、胜负判定的模块，必须覆盖边界时间与并发消息场景。
3. 未通过测试不得进入下一个模块开发。

## 19. 第一批可提交模块清单（直接进入 AI 编程）
以下模块按“低耦合 + 高价值 + 可快速闭环”原则排列。每个模块完成后都必须执行：`模块单测通过 -> commit`。

### 19.1 M1-01 棋盘基础模型（Board + Piece + Position）
**目标**：建立可被规则引擎复用的统一数据结构。  
**涉及范围**：`domain.board`  
**完成标准（DoD）**：
1. 能初始化标准中国象棋开局局面。
2. 支持按坐标读取、移动、回滚棋子。
3. 坐标转换规则唯一（例如 `a0` 到 `i9`）。

**单元测试最小集**：
1. 标准开局棋子数量和位置断言。
2. 空位/越界坐标访问异常断言。
3. 一步移动后局面一致性断言。

**建议 commit message**：
- `feat(board): add xiangqi board and position model`

### 19.2 M1-02 基础合法性框架（回合与通用校验）
**目标**：先实现“非棋子特定”的通用校验。  
**涉及范围**：`domain.rule`、`domain.state`  
**完成标准（DoD）**：
1. 仅允许当前回合方走子。
2. 目标格为己方棋子时拒绝。
3. 基础请求参数校验（from/to 不为空、不越界）。

**单元测试最小集**：
1. 走对方棋子必须失败。
2. 吃己方棋子必须失败。
3. 非法坐标必须失败。

**建议 commit message**：
- `feat(rule): add generic move validation pipeline`

### 19.3 M1-03 车/马走法校验
**目标**：实现两类高频棋子规则，打通规则引擎最小闭环。  
**涉及范围**：`domain.rule.piece`  
**完成标准（DoD）**：
1. 车走直线，路径不可有阻挡。
2. 马走“日”字，蹩马腿判定正确。

**单元测试最小集**：
1. 车直线无阻挡合法、有阻挡非法。
2. 马标准走法合法、蹩马腿非法。
3. 非法位移（如车斜走）非法。

**建议 commit message**：
- `feat(rule): implement rook and knight movement rules`

### 19.4 M1-04 炮走法校验
**目标**：实现最容易出错的炮架规则。  
**完成标准（DoD）**：
1. 炮平移与车一致（不吃子时路径无阻挡）。
2. 炮吃子必须隔一个子（炮架）。

**单元测试最小集**：
1. 不吃子时直线无阻挡合法。
2. 吃子时隔一个子合法。
3. 吃子时隔 0/2 个子非法。

**建议 commit message**：
- `feat(rule): implement cannon movement and capture rule`

### 19.5 M1-05 将/士/象/兵走法校验
**目标**：补齐剩余棋子规则。  
**完成标准（DoD）**：
1. 将在九宫内走，士斜走九宫，象不过河且塞象眼判定正确。

3. 发布包命名统一：`chuhanai-stage1-vX.Y.Z`。

### 31.2 回滚策略
1. 服务端保留最近两个稳定版本，可快速切回。
2. 客户端发布失败时提供上一稳定版本下载入口。
3. 重大故障（规则判定错误、无法匹配）触发紧急回滚。

## 32. 监控与告警（最小可用）
### 32.1 关键指标
1. 匹配成功率（5 分钟窗口）
2. 平均匹配耗时与 P95
3. 对局中断率（断线未恢复）
4. 走子请求失败率（含非法与系统错误分开统计）
5. 服务端心跳超时连接数

### 32.2 告警阈值建议
1. 匹配成功率低于 95% 持续 10 分钟告警。
2. 走子系统错误率高于 1% 持续 5 分钟告警。
3. 对局中断率高于 3% 持续 10 分钟告警。

### 32.3 日志最小规范
1. 每条关键消息记录 `msgId/roomId/sessionId/type/result`。
2. 对局结束必须记录 `endReason/winnerSide/totalMoveCount`。
3. 错误日志需要可关联到具体房间与消息 ID。
